# S3 Object Storage

The operator can use any S3-compatible object store for two independent purposes:

- **Log storage** — retains the last run's log output per project, queryable through the UI (alternative to the in-memory or Valkey store)
- **Renovate repository cache** — forwards S3 credentials and a cache URL to each Renovate executor job, allowing Renovate to reuse dependency metadata between runs

Both use the same bucket and credentials. They are separated by configurable key prefixes so they never collide.

## Configuration

S3 is configured via environment variables (or the equivalent Helm values):

| Environment variable        | Helm value                       | Default           | Description                                                                                  |
|-----------------------------|----------------------------------|-------------------|----------------------------------------------------------------------------------------------|
| `S3_BUCKET`                 | `s3.bucket`                      | `""`              | S3 bucket name. Required when using S3 log storage or S3 cache forwarding.                  |
| `S3_REGION`                 | `s3.region`                      | `us-east-1`       | AWS region.                                                                                  |
| `S3_ENDPOINT`               | `s3.endpoint`                    | `""`              | Custom endpoint URL for S3-compatible self-hosted stores (e.g. RustFS). Leave empty for AWS. |
| `S3_ACCESS_KEY_ID`          | via `s3.existingSecret`          | `""`              | Static access key. When empty, the pod's IAM role or default credential chain is used.      |
| `S3_SECRET_ACCESS_KEY`      | via `s3.existingSecret`          | `""`              | Static secret key.                                                                           |
| `S3_CREDENTIALS_SECRET_NAME`| via `s3.existingSecret`          | `""`              | Name of the K8s Secret holding credentials — set automatically by the Helm chart.           |
| `S3_LOG_PREFIX`             | `s3.logPrefix`                   | `renovate-logs`   | Object key prefix for the operator log store.                                                |
| `S3_CACHE_PREFIX`           | `s3.cachePrefix`                 | `renovate-cache`  | Object key prefix forwarded to Renovate jobs.                                                |
| `LOG_STORE_MODE`            | `config.logStorage.mode`         | `disabled`        | Log storage backend: `disabled`, `memory`, `valkey`, or `s3`.                               |
| `S3_FORWARD_CACHE_TO_JOBS`  | `config.forwardS3CacheToJobs`    | `true`            | Forward S3 cache configuration to Renovate executor jobs. Requires `S3_BUCKET` to be set.  |

## Object layout

All objects share the same bucket. The key structure is:

```
{logPrefix}/{namespace}/{renovateJobName}/{org/repo}.log   ← operator log store
{cachePrefix}/...                                          ← Renovate repository cache
```

The Renovate cache subtree is managed entirely by Renovate itself; the operator only sets the prefix root via `RENOVATE_REPOSITORY_CACHE_TYPE`.

## Credentials

### IAM role (recommended for AWS)

When running on AWS, attach an IAM role to the operator's service account (IRSA) or node group. Leave `s3.existingSecret` empty — the operator and Renovate jobs both pick up credentials from the default AWS credential chain automatically.

The IAM policy needs `s3:GetObject`, `s3:PutObject`, and `s3:ListBucket` on the bucket:

```json
{
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject", "s3:ListBucket"],
  "Resource": [
    "arn:aws:s3:::your-bucket",
    "arn:aws:s3:::your-bucket/*"
  ]
}
```

### Static credentials

Create a Kubernetes Secret with keys `access-key-id` and `secret-access-key`:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-s3-credentials
stringData:
  access-key-id: "AKIAIOSFODNN7EXAMPLE"
  secret-access-key: "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
```

Reference it in your Helm values:

```yaml
s3:
  existingSecret: my-s3-credentials
```

The Helm chart mounts the credentials onto the operator pod and passes the secret name to job pods so they can reference it directly — no intermediate secret is created.

## Helm examples

### Log storage only (AWS with IAM role)

```yaml
s3:
  bucket: my-renovate-bucket
  region: eu-west-1

config:
  logStorage:
    mode: s3
  forwardS3CacheToJobs: false
```

### Log storage and Renovate cache forwarding

```yaml
s3:
  bucket: my-renovate-bucket
  region: eu-west-1
  existingSecret: my-s3-credentials

config:
  logStorage:
    mode: s3
  forwardS3CacheToJobs: true
```

### S3-compatible self-hosted store (e.g. RustFS)

```yaml
s3:
  bucket: renovate
  region: us-east-1        # any non-empty value; most S3-compatible stores ignore it
  endpoint: https://rustfs.example.com
  existingSecret: my-s3-credentials

config:
  logStorage:
    mode: s3
  forwardS3CacheToJobs: true
```

Path-style addressing (`RENOVATE_S3_PATH_STYLE=true`) is enabled automatically when a custom endpoint is set.

## What gets forwarded to Renovate jobs

When `config.forwardS3CacheToJobs: true` and `s3.bucket` is set, executor and discovery jobs receive the following environment variables:

| Variable                       | Value                                    |
|--------------------------------|------------------------------------------|
| `RENOVATE_REPOSITORY_CACHE`    | `enabled`                                |
| `RENOVATE_REPOSITORY_CACHE_TYPE` | `s3://{bucket}/{cachePrefix}`          |
| `AWS_REGION`                   | value of `s3.region`                     |
| `RENOVATE_S3_ENDPOINT`         | value of `s3.endpoint` (if set)          |
| `RENOVATE_S3_PATH_STYLE`       | `true` (if `s3.endpoint` is set)         |
| `AWS_ACCESS_KEY_ID`            | from `s3.existingSecret` (if set)        |
| `AWS_SECRET_ACCESS_KEY`        | from `s3.existingSecret` (if set)        |
